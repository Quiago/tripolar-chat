Actualmente, nuestra plataforma FactoryMind AI tiene un problema de "Cold Start". El proceso de vLLM (a través de `VLLMModelManager`) se está instanciando y cargando el modelo en la GPU de forma perezosa (lazy) en el endpoint `/v1/chat/completions` la primera vez que un usuario envía un mensaje. Esto causa una demora inaceptable de ~60 segundos en el primer request.



Necesito que refactorices el código para aplicar las siguientes mejoras siguiendo estrictamente buenas prácticas, tipado fuerte y sin hardcodear nada (todo debe venir de `config.py` y `.env`):



1. **Configuración del Modelo por Defecto:**

   - Añade una variable `DEFAULT_MODEL` en `server/config.py` (tipo string). En `.env.example`, ponlo como `llama-3.1-8b-awq`.



2. **Warm-up en el Arranque del Servidor (FastAPI Lifespan):**

   - Modifica `server/main.py` para usar el context manager `@asynccontextmanager def lifespan(app: FastAPI):`.

   - Durante el arranque (antes del `yield`), el servidor debe instanciar el `VLLMModelManager` y llamar a `ensure_loaded(settings.default_model)`. De esta forma, el modelo por defecto se carga en la GPU, compila los grafos de CUDA y hace el warm-up ANTES de que el servidor acepte peticiones HTTP.

   - Asegúrate de manejar correctamente el apagado (`stop()`) de vLLM después del `yield` para liberar la GPU cuando el servidor se apague.



3. **Pre-descarga de Pesos (Opcional en Background):**

   - Añade una variable en `config.py` llamada `PREFETCH_ALL_MODELS` (booleano, por defecto `True`).

   - Si es `True`, añade un método en `VLLMModelManager` (ej. `prefetch_weights()`) que se ejecute en el `lifespan` y use `huggingface_hub.snapshot_download` para descargar a la carpeta de caché los pesos de todos los modelos definidos en `MODEL_CATALOG` si es que ya no está descargados en disco, verifica siempre que esten en disco antes de descargar. 

   - IMPORTANTE: Esto solo debe descargar los archivos al disco, NO cargarlos en la GPU (para no exceder la VRAM).



4. **Reglas:**

   - No rompas el patrón Singleton de `get_vllm_manager()`.

   - Maneja los posibles errores de timeout en el arranque (si vLLM falla al iniciar, el servidor FastAPI debería detenerse limpiamente e imprimir el error).

   - Mantén todo asíncrono donde sea apropiado para no bloquear el Event Loop de FastAPI.
